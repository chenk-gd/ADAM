**版本**: 1.8  
**日期**: 2026-05-08  
**状态**: 草案

## 1. 文档概述

### 1.1 目的
本文档详细描述研发资产管理系统的功能需求、非功能需求、数据模型及系统边界，为系统设计、开发和测试提供依据。

### 1.2 范围
本系统是一个为大型语言模型（LLM）提供研发资产上下文管理的服务平台，支持需求、设计文档、代码、测试用例等研发资产的全生命周期管理。

### 1.3 术语定义

| 术语          | 定义                                  |
| ----------- | ----------------------------------- |
| 研发资产        | 软件开发过程中产生的各类工作产物，包括需求、设计文档、代码、测试用例等 |
| 资产类型        | 研发资产的分类，如需求、工作项、架构设计文档等             |
| 资产实例        | 某一类型资产的具体实体，如"用户登录功能需求"是一个需求类型的实例   |
| 虚拟实例        | 临时构建的查询上下文，用于查询某类资产应依赖的上游资产，用完即弃    |
| 版本号         | 资产发布时指定的标识，由平台智能建议、人工确认               |
| 发布依赖快照      | 资产发布时声明的上游资产及版本，用于历史追溯和审计             |
| 当前有效依赖基线    | 经发布或审查确认后的当前上游版本，用于上下文查询和 Dirty 判断      |
| Clean 状态    | 资产内容可信，当前有效依赖基线已追平上游最新版本，或上游变化已审查确认无影响 |
| Dirty 状态    | 上游资产已发布新版本，但当前资产尚未通过重新发布或手工 Clean 处理    |
| Archived 状态 | 资产已停止维护，不再接收 Dirty，不允许发布新版本，但仍可被查询和追溯   |
| 项目级资产      | 属于特定项目的资产，如某功能的需求、代码提交等               |
| 组织级资产      | 跨项目共享的资产，如组织规范、编码标准、通用设计等             |
| 项目          | 资产的归属单位，资产可属于某个项目或属于组织（项目级/组织级）   |

### 2.1 系统目标

## 3. 功能需求

### 3.1 资产类型管理

#### FR-001: 资产类型定义
**描述**: 系统支持定义多种研发资产类型  
**优先级**: 高  
**详细说明**:
- 支持预定义资产类型：需求、工作项、概要设计、详细设计、代码提交、测试用例、流水线
- 支持自定义扩展资产类型
- 每种资产类型包含：ID、名称、描述、元数据schema

#### FR-002: 资产类型元数据
**描述**: 每种资产类型可定义元数据结构  
**优先级**: 中  
**详细说明**:
- 支持JSON Schema定义元数据字段
- 支持字段类型：字符串、数字、日期、枚举、布尔值
- 支持必填字段和可选字段配置

### 3.2 资产类型间依赖关系（预设依赖）

#### FR-003: 依赖关系配置
**描述**: 系统支持配置资产类型间的预设依赖关系  
**优先级**: 高  
**详细说明**:
- 定义哪些资产类型之间存在依赖关系
- 依赖关系是单向的：A 依赖于 B
- 依赖关系具有传递性（可选配置）
- **预设依赖关系必须构成 DAG**: 资产类型间的依赖关系不能形成循环
- 支持依赖关系的增删改查

**预设依赖关系示例**:

```
代码提交 ──────┬───> 工作项
              ├───> 详细设计文档
              └───> 需求

详细设计文档 ───> 概要设计文档
             ───> 需求

工作项 ─────────> 需求

测试用例 ───────> 需求
              ───> 详细设计

流水线 ─────────> 代码提交
```

#### FR-004: 依赖关系验证
**描述**: 创建资产实例时验证依赖关系的合法性  
**优先级**: 中  
**详细说明**:
- 只允许在预设依赖关系范围内建立实例依赖
- 非法依赖关系应被拒绝并返回错误

### 3.3 资产实例管理

#### FR-005: 资产实例创建
**描述**: 支持创建资产实例（首次发布时）  
**优先级**: 高  
**详细说明**:
- 资产实例在**首次发布时**由外部系统调用接口创建
- 实例包含属性：
  - **基础属性**: ID、类型、名称
  - **版本属性**: 当前版本号（首次发布时确定）
  - **来源属性**: source（git/wiki/jira等）、外部引用地址
  - **人员属性**: 发布人、责任人列表（可多责任人）
  - **状态属性**: 当前状态（首次发布为 Clean）
  - **元数据**: 按资产类型schema存储的自定义属性
  - **依赖关系**: 本次发布所依赖的上游资产及版本
- 支持创建虚拟实例用于依赖上下文查询

#### FR-006: 资产实例查询
**描述**: 支持查询资产实例及其历史版本  
**优先级**: 高  
**详细说明**:
- 支持按类型、状态、时间范围查询
- 支持分页和排序
- 支持查询指定版本的实例

#### FR-007: 资产实例更新
**描述**: 支持更新资产实例（重新发布时）  
**优先级**: 高  
**详细说明**:
- 资产重新发布时创建新版本
- 发布时指定版本号（平台智能建议、人工确认）
- 记录发布历史和发布说明
- 发布后状态恢复为 Clean（依赖已更新到最新）

#### FR-008: 资产实例删除
**描述**: 支持删除资产实例  
**优先级**: 中  
**详细说明**:
- 删除前检查是否存在下游依赖
- 存在依赖时禁止删除或提示确认
- 支持软删除（标记为删除状态）

### 3.4 实例间依赖关系

#### FR-009: 依赖关系建立
**描述**: 支持在发布时建立资产实例间的依赖关系  
**优先级**: 高  
**详细说明**:
- 依赖关系在**资产发布时**确定并固化
- 发布时指定该版本依赖的上游资产及其版本
- 实例依赖必须符合预设的资产类型依赖关系
- 一个版本可以依赖多个上游资产
- 发布依赖快照随版本保存，用于历史追溯和审计
- 发布时同步建立当前有效依赖基线，用于上下文查询和后续 Dirty 判断
- **依赖关系必须构成严格 DAG**: 禁止循环依赖（包括直接和间接循环）

#### FR-010: 依赖关系查询
**描述**: 支持查询资产的依赖关系图（DAG）  
**优先级**: 高  
**详细说明**:
- 查询指定实例的上游依赖（直接和间接）
- 查询指定实例的下游依赖（直接和间接）
- 可视化展示依赖关系图（严格DAG结构）

#### FR-011: 依赖关系变更
**描述**: 支持修改资产实例的依赖关系  
**优先级**: 中  
**详细说明**:
- 已发布版本的发布依赖快照不可原地修改
- 添加或移除依赖关系应通过重新发布新版本完成
- 手工 Clean 仅允许更新当前有效依赖基线，不修改发布依赖快照
- 变更当前有效依赖基线时触发自身状态重计算，不触发下游 Dirty

### 3.5 版本管理

#### FR-012: 版本发布管理
**描述**: 资产通过发布操作纳入平台管理，版本号由平台智能建议、人工确认  
**优先级**: 高  
**详细说明**:
- **发布触发**: 仅当外部系统显式调用发布接口时，平台才创建新版本
- **版本号生成**:
  - 平台根据变更内容智能建议版本类型（major/minor/patch）
  - 用户确认或修改后生效
  - 支持语义化版本（SemVer）
- **发布时固化**:
  - 发布时确定该版本依赖的上游资产及版本
  - 依赖关系随版本快照保存，后续上游变更不影响历史版本
  - 发布完成后，将当前有效依赖基线更新为本次发布声明的上游版本
- **首次发布**: 资产实例首次发布时创建，版本号从 v0.1 或 v1.0 开始

**版本号建议规则示例**:
```
变更内容分析          建议版本类型    当前版本 -> 建议版本
重大功能变更/破坏性修改   major        v1.2.3 -> v2.0.0
新增功能（向后兼容）     minor        v1.2.3 -> v1.3.0
问题修复/小幅调整       patch        v1.2.3 -> v1.2.4
```

#### FR-013: 版本历史与对比
**描述**: 支持查看资产的发布历史和版本间对比  
**优先级**: 中  
**详细说明**:
- 查询资产的所有发布版本
- 对比两个发布版本的元数据差异
- 对比两个发布版本的依赖关系差异
- 显示发布时间、发布人、发布说明

### 3.6 状态管理

#### FR-015: 资产状态定义
**描述**: 定义资产实例的状态体系  
**优先级**: 高  
**详细说明**:
- **Clean**: 当前资产版本可信；其当前有效依赖基线已追平上游最新发布版本，或已通过审查确认上游变化不影响该资产
- **Dirty**: 当前资产存在至少一个上游资产已发布新版本，但尚未通过重新发布或手工 Clean 处理
- **Archived**: 资产已停止维护，不再接收 Dirty，不允许发布新版本；仍保留在依赖图中用于历史追溯和查询
- 平台只管理正式发布后的资产版本及依赖关系；源平台中的草稿、编辑中、PR、评审中、构建中等未发布状态不进入本平台状态机

#### FR-016: 状态自动转换
**描述**: 资产状态根据依赖关系自动转换  
**优先级**: 高  
**详细说明**:
- 当上游依赖资产发布新版本时，直接下游资产状态变为 Dirty
- 资产重新发布后（依赖已更新到最新版本），状态恢复为 Clean，并触发直接下游资产 Dirty
- 支持手工 Clean：经审查确认上游变化不影响该资产时，更新该资产的当前有效依赖基线至已审查通过的上游最新版本，记录审查确认日志，状态恢复为 Clean
- 手工 Clean 不创建新资产版本，不触发下游 Dirty
- 手动标记资产为 Archived
- 提供批量状态刷新功能

**状态转换图**:
```
           ┌─────────────┐
     ┌────>│   Clean     │<────┐
     │     └─────────────┘     │
     │           │             │
依赖发布│           │ 重新发布/手工Clean
     │           ▼             │
     │     ┌─────────────┐     │
     └─────┤   Dirty     │─────┘
           └─────────────┘
                 │
                 │ 手动归档
                 ▼
           ┌─────────────┐
           │  Archived   │
           └─────────────┘
```

#### FR-017: 状态传播规则
**描述**: 定义状态变更的传播规则  
**优先级**: 高  
**详细说明**:
- **核心原则**: 只有资产发布新版本才会触发下游 Dirty；状态本身的变更不会传播
- **触发条件**: 资产发布新版本后，基于当前有效依赖基线查找直接下游资产，并将其标记为 Dirty
- **传播方向**: 发布事件从被依赖资产传播到直接下游资产
- **变更逻辑**: 
  - 上游资产发布新版本 → 所有以该资产旧版本作为当前有效依赖基线的直接下游资产状态变为 Dirty
  - 下游资产发布新版本（更新依赖到最新上游版本）→ 自身状态恢复 Clean，并作为发布事件触发其直接下游 Dirty
  - 下游资产手工 Clean（确认上游变化无影响）→ 更新自身当前有效依赖基线并恢复 Clean，不触发其下游 Dirty
  - 资产归档 → 不触发下游 Dirty，不改变下游 Clean/Dirty 状态；依赖图保留归档节点并在查询结果中标记
- **传播范围**: 
  - 直接依赖的资产必定受发布事件影响
  - 间接依赖的资产不会由同一次发布事件直接标记 Dirty；只有中间资产重新发布新版本时，才继续触发其下游 Dirty
- **并发与队列**:
  - 多个上游同时发布导致同一资产 Dirty 时，合并 Dirty 来源，保留每个上游的最新待处理版本
  - Dirty 处理中上游再次发布时，更新 Dirty 来源的目标版本并累计影响等级
  - 批量发布支持事务性处理，中间状态为 publishing，全部成功后统一触发直接下游 Dirty
- **Dirty 队列优先级**:
  - 按影响等级、等待时长、资产类型权重计算优先级
  - 等待时间越长优先级越高，避免长期积压
  - 示例公式：`score = impact_level_weight + min(hours_waiting, 72) * age_weight + asset_type_weight`

**传播示例**:
```
初始状态：
  A v1.0(clean) -> B v1.0(clean, effective A v1.0) -> C v1.0(clean, effective B v1.0)

场景1：B 主动发布 v1.1
  A v1.0(clean) -> B v1.1(clean, effective A v1.0) -> C v1.0(dirty, B v1.0 -> B v1.1)
  说明：B 发布新版本，直接下游 C 变为 Dirty。

场景2：A 发布 v1.1
  A v1.1(clean) -> B v1.1(dirty, A v1.0 -> A v1.1) -> C v1.0(dirty, B v1.0 -> B v1.1)
  说明：A 发布新版本，直接下游 B 变为 Dirty；C 保持 Dirty，不因 B 状态变化新增传播。

场景3：B 手工 Clean
  A v1.1(clean) -> B v1.1(clean, effective A v1.1) -> C v1.0(dirty, B v1.0 -> B v1.1)
  说明：B 经审查确认 A v1.1 不影响自身，更新当前有效依赖基线并恢复 Clean；不触发 C 新增 Dirty。

场景4：C 发布 v1.1
  A v1.1(clean) -> B v1.1(clean, effective A v1.1) -> C v1.1(clean, effective B v1.1)
  说明：C 发布新版本后更新依赖基线并恢复 Clean。
```

### 3.7 外部系统集成

#### FR-018: 外部内容存储
**描述**: 系统不存储实际内容，依赖外部系统  
**优先级**: 高  
**详细说明**:
- 资产实例只存储元数据和外部引用地址
- 实际内容存储于外部系统：
  - 代码 → Git 平台
  - 文档 → Wiki/文档管理系统
  - 工作项 → 项目管理系统

#### FR-019: 外部系统接入
**描述**: 支持外部系统接入资产管理平台  
**优先级**: 高  
**详细说明**:
- 提供标准接口供外部系统调用
- 支持注册资产到平台
- 支持查询资产元数据
- 支持获取资产内容（代理转发或重定向）

#### FR-019A: Git 自动化集成（推荐）
**描述**: 支持通过 Git Hooks 在代码提交时自动注册资产和建立依赖关系  
**优先级**: 高  
**详细说明**:
- **集成方式**:
  - **Git Hooks**（推荐）: `post-commit` 或 `post-push` 钩子
  - **CI/CD Pipeline**: GitHub Actions、GitLab CI、Jenkins 等
  - **Webhook**: Git 平台推送事件监听
  
- **自动注册流程**:
  ```
  1. 开发人员本地 commit
  2. Git Hook / CI / Webhook 触发
  3. 解析 commit message 获取关联工作项（如 "Fix #123"）
  4. 调用平台 API 创建/更新代码提交资产
  5. 自动建立依赖关系（代码提交 → 工作项）
  6. 触发版本发布（可选）
  ```

- **发布边界与幂等**:
  - `post-commit` 默认仅用于本地预检、依赖关系预览或 dry-run，不直接注册正式资产
  - 正式注册建议以 `post-push`、PR/MR 合并、CI 成功或 Git 平台 Webhook 为准
  - 代码提交资产使用 `source + project_id + repository + commit_sha` 作为幂等键，避免 Hook、CI、Webhook 重复注册
  - Commit 资产的版本标识优先使用 commit SHA、tag 或 release 名称；只有显式发布场景才使用 SemVer 建议
  - 自动发布必须显式配置，且必须满足依赖合法性、项目边界和 DAG 校验
  
- **Commit Message 规范**:
  ```
  type: description
  
  Refs: WORK-123, DESIGN-456  # 关联上游资产ID
  Release: minor               # 可选：触发自动发布
  ```
  
- **依赖自动识别**:
  - 从 commit message 提取关联的工作项ID
  - 从分支名提取特性ID（如 `feature/USER-123-login`）
  - 从 PR/MR 描述提取关联需求
  
- **CLI 工具支持**:
  - 提供命令行工具简化 Hook 安装和配置
  - 支持本地预览（dry-run 模式）
  
**示例配置**:
```bash
# 安装 git hook
asset-cli hook install --type=post-commit --project=proj-123

# commit message 示例
feat: implement user login

Refs: WORK-123, DESIGN-456
Release: minor
```

#### FR-019B: CI/CD 流水线集成
**描述**: 支持流水线执行后自动注册为资产，并提供保留策略控制数据量  
**优先级**: 中  
**详细说明**:
- **资产范围**:
  - 流水线定义作为资产类型，可按配置发布版本
  - 每次流水线执行作为 PipelineRun（流水线执行记录），不等同于流水线资产的新发布版本
  - PipelineRun 可关联代码提交、测试报告、构建产物等资产，但默认不参与 Dirty 状态传播
- **集成触发**: CI/CD 平台通过 Webhook 或 Post-action 调用平台 API
- **注册内容**:
  - 流水线元数据：执行ID、触发分支、提交SHA、执行时间、持续时间
  - 执行结果：成功/失败/取消、测试报告摘要、构建产物链接
  - 日志摘要：错误日志片段、关键步骤日志（非完整日志）
  
- **数据量控制策略**:
  - **保留策略可配置**（参见 FR-027 通用保留策略管理）:
    - 最近 N 次执行（如保留最近 50 次）
    - 最近 M 天（如保留最近 30 天）
    - 保留特定版本（如发布成功的版本永久保留）
    - 失败版本单独保留策略（失败通常更重要，可保留更多）
  - **日志存储分级**:
    - **平台存储**: 执行摘要、结果状态、关键错误片段（< 10KB）
    - **外部存储**: 完整日志存储于 CI/CD 平台或对象存储，平台仅保存链接
  - **追溯约束**: 被资产版本、依赖快照、Dirty 处理日志引用的 `PipelineRun` 摘要不得物理删除
    
- **使用场景**:
  - AI Agent 排查故障：查询最近失败的流水线，获取错误日志
  - 关联代码提交：流水线失败 → 查找对应代码提交 → 分析影响范围
  - 质量报告：统计成功率、失败趋势

**示例配置**:
```yaml
# 流水线保留策略配置
pipeline_retention:
  default:
    count: 50          # 保留最近50次
    days: 30           # 或保留最近30天
  success:
    count: 20          # 成功版本保留较少
  failed:
    count: 100         # 失败版本保留更多（便于排查）
    days: 90
  tagged:
    permanent: true    # 标签版本永久保留
```

#### FR-027: 版本保留策略管理
**描述**: 支持为不同资产类型配置版本保留策略，控制存储成本和数据生命周期  
**优先级**: 中  
**详细说明**:
- **策略配置层级**:
  - **组织级默认**: 每种资产类型的全局默认策略
  - **项目级覆盖**: 特定项目可覆盖默认策略
  
- **保留维度**:
  - **按数量**: 保留最近 N 个版本
  - **按时间**: 保留最近 M 天内的版本
  - **按状态**: 不同状态不同策略（如失败版本保留更多）
  - **永久保留**: 特定条件下的版本永久保留
  
- **策略组合规则**:
  - 同时满足多个条件（如保留最近 50 次且不超过 30 天）
  - 按状态分层（失败版本单独配置）
  - 标签/标记版本特殊处理
  
- **清理机制**:
  - 自动定期清理（如每日凌晨）
  - 标记为归档而非物理删除（可配置）
  - 清理前发送通知

- **不可破坏追溯约束**:
  - 资产版本元数据、发布依赖快照、当前有效依赖基线、Dirty 处理日志默认永久保留
  - 仍被 `declared_version`、`effective_version`、版本快照、Dirty 处理日志或审计记录引用的版本，不得物理删除
  - 可清理的是外部内容缓存、完整日志、构建产物链接等大体积附属数据；清理后必须保留可解释摘要和外部引用状态
  - `Archived` 是资产生命周期状态，保留策略中的归档/冷存储是存储策略，两者不能混用
  - 对无法保留完整内容的版本，至少保留版本号、发布时间、发布人、元数据摘要、依赖快照和清理原因

**按资产类型推荐策略**:

| 资产类型 | 产生频率 | 推荐策略 | 说明 |
|----------|----------|----------|------|
| 代码提交 | 高频 | 最近 100 次 / 6 个月 | 版本多但 diff 较小 |
| 文档 | 低频 | 永久保留 | 重要且体积不大 |
| 工作项 | 中频 | 永久保留 | 业务重要 |
| 流水线 | 高频 | 最近 50 次 / 30 天 | 日志体积大 |
| 测试报告 | 高频 | 最近 30 次 / 14 天 | 报告体积大 |
| 设计文档 | 低频 | 永久保留 | 重要参考资料 |

**示例配置**:
```yaml
# AssetType 保留策略配置
retention_policies:
  code_commit:
    default:
      count: 100
      days: 180
    tagged:
      permanent: true
      
  document:
    default:
      permanent: true
      
  pipeline:
    default:
      count: 50
      days: 30
    failed:
      count: 100
      days: 90
    tagged:
      permanent: true
      
  work_item:
    default:
      permanent: true
    archived_storage:
      tier: cold   # 归档资产可转冷存储，但版本元数据和依赖快照仍永久保留
```

### 3.9 虚拟实例（查询上下文构建器）

#### FR-022: 虚拟资产实例支持
**描述**: 支持创建临时虚拟实例，用于构建查询上下文以查找某类资产应依赖的上游资产  
**优先级**: 高  
**详细说明**:
- **核心用途**: 
  - AI Agent 需要根据某工作项生成代码时，创建虚拟"代码提交"实例
  - 通过虚拟实例查询该类型资产（代码提交）应依赖的上游资产
  - 获取上下文：工作项本身、设计文档、编码规范等
  - 查询完成后虚拟实例即废弃
  
- **特性**:
  - 不对应任何实际资产，仅作为查询载体
  - 指定目标资产类型（如 "code_commit"）
  - 可临时指定一个或多个上游资产作为锚点
  - 基于资产类型预设依赖规则，推导完整的依赖上下文
  
- **生命周期**:
  - 临时创建（可设置短期过期时间）
  - 仅用于查询操作，不产生持久化数据
  - 实际代码提交到平台后，创建真实实例替代

**使用流程示例**:
```
1. AI Agent 获取工作项A（需要为此编写代码）
2. 创建虚拟实例：类型="code_commit"，锚点=[工作项A]
3. 查询虚拟实例的依赖上下文 → 返回：工作项A、详细设计文档、编码规范
4. AI Agent 基于上下文生成代码
5. 代码提交到 Git → 创建真实代码提交实例，建立真实依赖关系
6. 虚拟实例过期自动清理（或立即删除）
```

### 3.10 项目与组织层级

#### FR-026: 项目级与组织级资产管理
**描述**: 资产分为项目级和组织级，支持按项目查询并自动包含组织级资产  
**优先级**: 高  
**详细说明**:
- **资产层级**:
  - **项目级资产**: 属于特定项目，如某项目的需求、代码提交、测试用例等
  - **组织级资产**: 跨项目共享，如组织编码规范、通用设计规范、共享组件等
  
- **查询规则**:
  - 查询时必须指定项目ID
  - 返回结果包含：该项目下的所有项目级资产 + 组织级资产
  - 组织级资产在项目中可见但不可修改（由组织管理员维护）

- **依赖边界**:
  - 资产依赖关系不得跨组织建立
  - 项目级资产不得依赖其他项目的项目级资产
  - 组织级资产不得依赖项目级资产
  - 项目级资产可在上下文查询中引用组织级资产，但该引用不建立 Dirty 传播关系
  - 组织级资产发布新版本不向项目级资产传播 Dirty；项目资产需要更新组织级上下文时，通过查询或手工审查处理
  
- **使用场景**:
  - AI Agent 获取某项目工作项的上下文时，自动包含组织级编码规范
  - 查询某项目依赖时，可同时看到项目设计和组织标准

- **权限控制**:
  - 项目级资产：仅项目成员可见
  - 组织级资产：全员可见
  - 影响分析不得返回其他项目或其他组织的资产明细

## 7. 业务规则

### BR-001: 依赖合法性
资产实例只能依赖于符合预设依赖规则的类型。例如：代码提交实例不能依赖于测试用例实例。

### BR-002: 依赖关系必须构成严格 DAG
**依赖关系必须构成严格的有向无环图（Directed Acyclic Graph, DAG）**

- **禁止循环依赖**: 如果 A 依赖于 B，则 B 不能直接或间接依赖于 A
- **传递性检查**: 系统必须检查间接依赖关系，确保不存在隐式循环
- **验证时机**: 
  - 建立依赖关系时立即验证
  - 批量导入依赖时批量验证
  - 发现循环时拒绝操作并返回错误

**循环依赖检测示例**:
```
A ──> B ──> C ──> A  ❌ 循环依赖（C间接依赖A）

A ──> B ──> C        ✓ 合法DAG
└──> D
```

### BR-003: 状态传播触发条件
只有资产发布新版本时才触发直接下游资产状态变更为 Dirty。手工 Clean、归档、刷新状态等操作不触发下游 Dirty。

### BR-004: 版本号规则
- 版本号格式：遵循语义化版本（SemVer），如 v1.2.3
- 版本号生成：平台根据变更内容智能建议（major/minor/patch），人工确认后生效

### BR-005: 删除约束
存在下游依赖的资产实例不能被删除。必须先解除或转移所有下游依赖。

### BR-006: 归档约束
归档后的资产实例不能被修改，也不能再建立新的依赖关系。

### BR-007: 手工 Clean 约束
- 手工 Clean 必须基于审查确认，记录处理日志
- 手工 Clean 更新当前有效依赖基线，不修改发布时声明的依赖版本
- 手工 Clean 不创建新资产版本，不触发下游 Dirty
- 手工 Clean 后，后续 Dirty 判断以更新后的当前有效依赖基线为准

### BR-008: 依赖边界约束
- 依赖关系不得跨组织建立，也不得跨项目建立
- 项目级资产只能依赖同一项目内的项目级资产
- 组织级资产只能依赖同组织内的组织级资产，不得依赖项目级资产
- 项目上下文查询可合并返回组织级资产，但该合并结果不形成依赖关系，也不参与 Dirty 传播
- 影响分析仅在资产所属组织和项目边界内执行

### BR-009: 保留策略追溯约束
- 保留策略不得物理删除仍被依赖快照、当前有效依赖基线、Dirty 处理日志或审计记录引用的版本
- 版本元数据、依赖快照、Dirty 处理日志和审计摘要默认永久保留
- 对高频执行记录和日志，可清理完整内容，但必须保留可解释摘要、外部引用和清理原因

### BR-010: 自动化注册幂等约束
- 外部系统自动注册资产时必须提供幂等键
- Git 代码提交资产幂等键为 `source + project_id + repository + commit_sha`
- CI/CD 执行记录幂等键为 `source + project_id + pipeline_id + run_id`
- 自动化注册不得绕过依赖合法性、项目边界、组织边界和 DAG 校验
